# QAR構想チェック

## 目的

構造テンプレート適用（Phase 1: 骨格作成、`reference/講義テンプレート設計.md` 参照）の前に、講義の構想レベルでQARの整合性を検証する。キーフレーズ（大A）が講義タイトル（大Q）の答えになっているか、4つの小Aが大Aの根拠として成立するかを、ユーザーとの対話で確定する。

## 責務境界

- **qar-align**: QMDファイル作成前に `reference/2026_講義全体像.md` のテキストレベルで構想の整合性を検証する
- **verify-qmd**: QMDファイル上の `.key-phrase-box` テキストや構造の検証を担当する
- **適用除外**: 第13回（見学）・第14回（発表）は4セット構造を持たない。A1（大Aは大Qの答えか）のみ実行する

## 入力

- 対象の講義回（例: 第1回）
- 参照元: `reference/2026_講義全体像.md` の該当回セクション

## チェック項目

### L1: QAR整合

| # | チェック | 方法 | 合格基準 |
|:-:|:---|:---|:---|
| A1 | 大Aは大Qの答えか | 大Q（講義タイトル）を疑問文として読み、大A（キーフレーズ）がその答えになるか判定 | 「Q: [大Q]? → A: [大A]」と言って意味が通る |
| A2 | 小A×4は大Aの根拠か | 分解パターンを宣言し、小A①〜④を並べて大Aが導かれるか検証 | パターン名を宣言でき、4つの小Aから大Aへの論理的つながりが説明できる |
| A3 | 大Aと講義内容の一致 | 4セットの「扱う内容」を通読し、大Aがその全体を象徴しているか判定 | 大Aを読んだ人が「あ、こういう話をするんだな」と正しく予想できる |
| A4 | 小Aは小Qの答えか | 各小Q（疑問文）に対し小Aが直接の答えになるか判定 | 「Q: [小Q]? → A: [小A]」と言って意味が通る |
| A5 | 小Aは結論として成立するか | 「答えを先に言います。[小A]」と口に出してみる | 具体的で、聞いた学生が「へえ」「なるほど」と反応できる一文 |

### L2: 全体整合

| # | チェック | 方法 | 合格基準 |
|:-:|:---|:---|:---|
| B1 | セット順序の合理性 | 感情アーク（記載あれば）と照合。なければ教育的合理性（基礎→応用、概念→具体、既知→未知）で判定 | 順序を入れ替えると理解が損なわれる理由を説明できる |
| B2 | 大Aの分解可能性 | 大Aのあとに接続文（「これを4つの問いで掘り下げます」等）を置き、4つの小Qを列挙してみる | 接続文+4つの小Qが自然に聞こえる |
| B3 | インタラクション計画の実現可能性 | 講義全体像.mdの「共感」「活用」フィールドを読み、各セットで学生が考える/動く場面が計画されているか確認 | 共感（問いかけ）は全セットに記載がある。活動（投票/検索/挙手/計算等）は2セット以上に具体的な形式が明示されている |

## 分解パターンの類型

A2の判断基準。小A×4が以下のいずれかのパターン（混合可）に当てはまることを確認し、パターン名を宣言する。

| パターン | 説明 | 例 |
|:---|:---|:---|
| **側面分解** | 大Aの異なる側面を4つに分ける | 光・温度・水・構造 |
| **具体化分解** | 大Aの抽象的主張を4つの具体例で裏付ける | 身近さ・規模・先端・学ぶ意義 |
| **因果分解** | 大Aに至る論理ステップを4つに並べる | 原因→メカニズム→結果→対策 |
| **時系列分解** | 大Aを歴史的/プロセス的に4段階で追う | 過去→現在→近未来→展望 |
| **問題解決分解** | 問題提起→原理→応用→限界の流れ | 何が問題→なぜ→どう解決→残る課題 |

**混合型**: 4セットが単一パターンに収まらない場合、「主パターン: 具体化分解 + セット4はメタ視点」のように主パターン+例外を宣言する。純粋型でなくてもよいが、少なくとも主パターンが説明できること。

パターンが全く当てはまらない場合は大Aと小A群の不整合を疑い、ユーザーに確認する。

## 小A候補の導出ルール

Step 2で小A候補を導出する際の操作的定義:

1. `講義全体像.md` の「扱う内容」列を読む
2. 各セットの「扱う内容」から**結論となる一文**を抽出する
   - 「〜である」「〜だ」で終わる宣言文にする
   - 一文で完結し、30字以内を目安とする
   - 「扱う内容」に明示的結論がなければ、内容全体を要約した主張文を作る
3. 導出した小A候補を「答えを先に言います。[小A]」のテストに通す
4. テストに通らなければ言い回しを修正する

## 処理フロー

### Step 1: 情報抽出

`reference/2026_講義全体像.md` から対象回のセクションを読み、以下を抽出する:

- 大Q（講義タイトル）
- キーフレーズ（大A候補）
- 4つの小Q案 + 扱う内容
- 感情アーク/講師メモ（あれば）

### Step 2: L1チェック — QAR整合を分析

1. 大A候補が大Qに答えているか評価（A1）
2. 4つの小Qの「扱う内容」から**小A導出ルール**に従って小A候補を導出
3. 大Aが4セットの「扱う内容」全体を象徴しているか評価（A3）
4. 分解パターン類型に照合し、パターン名を宣言（A2）
5. 小A候補×4が大A候補の根拠になるか評価（A2）
6. 各小Aが各小Qの答えになっているか評価（A4）
7. 各小Aが「答えを先に言います」で提示できる具体性を持つか評価（A5）

### Step 3: 結果をユーザーに提示し、対話で確定する

**出力**: 出力フォーマットに従って整合表を提示する。

**対話フロー**:

1. **全項目✅の場合**: 「整合性に問題ありません。この内容で Phase 1 に進んでよいですか？」とユーザーに確認。ユーザーが「OK」と明示したら確定。
2. **⚠️がある場合**: 不整合箇所ごとに修正案を提示し、ユーザーの判断を仰ぐ。
   - **大Aの変更が必要な場合**: 大A修正案を提示 → ユーザー承認 → 小A×4との整合を再チェック（Step 2に戻る）
   - **小Aの個別修正で済む場合**: 該当小Aの修正案を提示 → ユーザー承認 → 大Aとの整合を再確認
   - **大A・小A両方の再構成が必要な場合**: 新しい大A候補+分解パターン+小A候補のセットを提案 → ユーザーと対話で練り上げる
3. **確定条件**: ユーザーが大A + 小A×4のセットに対して明示的に「OK」「確定」「これで進めて」等と承認した時点で確定とする。

### Step 4: L2チェック — 全体整合を確認

- セット順序の合理性を確認（B1）
- 大Aの分解可能性を確認（B2）
- インタラクション計画の実現可能性を確認（B3）
  - 「共感」フィールド: 具体的な問いかけが記載されているか
  - 「活用」フィールド: 活動形式（投票/検索/挙手/計算等）が明示されているか
  - 記載がないセットがあれば⚠️として指摘（第13-14回は適用除外）
- 問題があればStep 3に戻って対話

### Step 5: 確定出力

確定した大A + 小A×4を以下の形式で出力し、Phase 1への入力とする:

```markdown
### 確定QAR（第N回）
- **大A**: [確定キーフレーズ]
- **小A①**: [確定小A]
- **小A②**: [確定小A]
- **小A③**: [確定小A]
- **小A④**: [確定小A]
- **分解パターン**: [パターン名]
```

`講義全体像.md` の更新提案:
- **更新範囲**: キーフレーズの差し替え + セットテーブルへの「小A」列追加のみ
- 小Q案・扱う内容・講師メモは変更しない
- ユーザー承認後に実施
